## 배경 및 구현 목적

### 해결하려는 문제

현재 `SeasonalDecoder`는 `{daily:1d, weekly:7d, yearly:365.25d}` 3개 주기만 사용하며,
주기 전체를 binary on/off로 마스킹한다. 이로 인해:

1. **amplitude 표현 한계:** 복합 주기(주간/주중 amplitude 차이 등)를 충분히 학습하지 못함
2. **granularity별 커버리지 공백:** `{7.1d, 15.1d}` 범위 및 fine-grained 데이터에서 weekly 상위 주기가 전혀 활성화 안 됨
3. **일반화 부족:** 데이터 주기에 따라 외부에서 마스크를 주입하는 구조

### 개선 방향

- **주기 추가:** monthly(30.4375d)를 4번째 주기로 추가
- **per-harmonic masking:** 주기 단위 binary → 고조파(k) 단위 개별 활성화
- **1.5× 분리 원칙:** 인접 주기 간 주파수 경합 방지
- **소프트 마스킹으로의 전환 포석:** 이번 하드 마스크 구현이 이후 임베딩 조건부 소프트 마스킹의 기반

## 핵심 설계 원칙 (구현 시 반드시 준수)

### 1. 주기 및 오더 정책

```
PERIODS      = {daily: 1.0,  weekly: 7.0,  monthly: 30.4375, yearly: 365.25}  (단위: 일)
K_MAX        = {daily: 10,   weekly: 4,    monthly: 2,        yearly: 8}
```

1.5× 분리 원칙 검증:
```
weekly  k=4 finest: 7/4 = 1.75d  ÷  daily  1.0d = 1.75× ≥ 1.5 ✓ (현행 유지)
monthly k=2 finest: 30.4/2=15.2d ÷  weekly 7.0d = 2.17× ≥ 1.5 ✓
yearly  k=8 finest: 365.25/8=45.7d ÷ monthly 30.4d = 1.50× ≥ 1.5 ✓ (경계값)
yearly  k=12 = 365.25/12 = 30.4375d = monthly 기본파 → 정확 중복! k=8 고수 이유
```

**biweekly(14d) 별도 주기 없음.** monthly k=2(15.2d)가 그 구간을 커버.
gap [7.1d, 15.1d]은 의도적으로 허용 (잔차 디코더 + 이후 소프트 마스킹이 담당).

### 2. per-harmonic 마스킹 공식

```python
# 고조파 (P, k)의 활성화 조건
harmonic_period = P / k          # 단위: 일
fd = FREQ_DAYS[freq]             # 단위: 일/스텝
context_span = context_len * fd  # 단위: 일

active = (fd < harmonic_period) and (context_span >= harmonic_period)
```

현행 binary 마스킹과의 차이:
- **현행:** k=1 조건이 True이면 k=1..K_MAX 전체 활성화
- **변경:** 각 k마다 개별 조건 평가 → 일부 k만 활성화 가능

### 3. freq별 활성 고조파 (context_len=512 고정)

| freq | context_span | daily k=10 | weekly k=4 | monthly k=2 | yearly k=8 |
|---|---|---|---|---|---|
| 5-min (1/288) | 1.78d | 1~10 | k=4만 | — | — |
| 10-min (1/144) | 3.56d | 1~10 | k=2~4 | — | — |
| 30-min (1/48) | 10.67d | 1~10 | 1~4 | — | — |
| hourly (1/24) | 21.33d | 1~10 | 1~4 | **k=2** | — |
| daily (1.0) | 512d | — | 1~4 | **k=1~2** | 1~8 |
| weekly (7.0) | 3584d | — | — | 1~2 | 1~8 |
| monthly (30.4) | 15584d | — | — | — | 1~8 |

5-min에서 weekly k=4(1.75d)만 활성화되는 것은 의도된 동작.

## Phase 2 — Fourier 기저 재캐싱

### 공통 기저 생성 함수 (common.py)

```python
PERIODS = {"daily": 1.0, "weekly": 7.0, "monthly": 30.4375, "yearly": 365.25}
K_MAX   = {"daily": 10, "weekly": 4, "monthly": 2, "yearly": 8}

FREQ_DAYS = {
    "5_minutes": 1/288, "10_minutes": 1/144,
    "half_hourly": 1/48, "hourly": 1/24, "H": 1/24,
    "D": 1.0, "daily": 1.0,
    "weekly": 7.0, "monthly": 30.4375, "yearly": 365.25,
}

def build_fine_mask_basis(freq: str, context_len: int, horizon: int) -> dict:
    fd = FREQ_DAYS[freq]
    context_span = context_len * fd
    # 중요: t는 context_len에서 시작 (위상 정렬)
    t = torch.arange(context_len, context_len + horizon, dtype=torch.float32)

    result = {}
    for family, P in PERIODS.items():
        k_max = K_MAX[family]
        basis = torch.zeros(horizon, 2 * k_max, dtype=torch.float32)
        p_steps = P / fd
        for k in range(1, k_max + 1):
            harmonic_period = P / k
            if fd < harmonic_period and context_span >= harmonic_period:
                basis[:, 2*(k-1)]   = torch.sin(2 * math.pi * k * t / p_steps)
                basis[:, 2*(k-1)+1] = torch.cos(2 * math.pi * k * t / p_steps)
        result[f"{family}_basis"] = basis
    return result
```

### 알려진 기존 오류 및 수정 이력

1. **시간축 오류 (수정 완료):** 기저 생성 시 `t = arange(0, horizon)` → `t = arange(context_len, context_len + horizon)`
   - 발견 위치: 구버전 `prepare_cache.py`의 `build_fourier_basis()`
   - 영향: seasonal 위상 오정렬 (주파수는 맞으나 위치 틀림)
   - 5.30fine_mask 구현 시 반드시 새 공식 사용

2. **N_FOURIER_TERMS 혼용 (주의):** 기존 코드에서 HEAD 용량(10,10,10)과 실제 오더(10,4,8)가 다름
   - `decoder_seasonal.py`의 `N_FOURIER_TERMS` = HEAD 용량
   - `FOURIER_PROFILE` in 학습 스크립트 = 실제 사용 오더
   - 5.30fine_mask에서는 양쪽 동일하게 `{daily:10, weekly:4, monthly:2, yearly:8}` 통일

3. **`_pad_width()` 동작:** HEAD 용량보다 작은 basis가 들어오면 0으로 패딩
   - 기존 S1-S10 캐시의 weekly basis는 10 columns → 신규 HEAD(4 columns)로 자름(truncate)
   - `_pad_width()` 또는 `limit_or_pad_basis()` 에서 truncate 방향 확인 필요

4. **nonF 학습의 seasonal_coefficients:** nonF 합성 데이터는 Fourier 계수 타깃이 placeholder(0)
   - `train_nonfourier_finetune.py`는 계수 loss 없이 raw seasonal_n만 사용
   - 5.30fine_mask 학습에서도 동일 정책 유지
